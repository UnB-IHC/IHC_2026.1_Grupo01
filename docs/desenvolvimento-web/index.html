🛠️ Desenvolvimento Web

O que é?​

Análise técnica dos critérios de acessibilidade relacionados ao código, HTML, navegação e funcionalidade. Este guia estabelece os parâmetros técnicos necessários para garantir a conformidade do projeto com as normas de acessibilidade vigentes.

Construir essa infraestrutura acessível exige alinhar as práticas de codificação às principais diretrizes de referência: as normas internacionais da WCAG 2.2, a regulamentação técnica brasileira da NBR 17225 e as orientações práticas do Guia de Boas Práticas para Acessibilidade Digital do Gov.br.

1. Semântica e Estrutura do HTML​

O código-fonte deve usar tags HTML5 nativas para dar significado e hierarquia lógica ao conteúdo, permitindo que tecnologias assistivas entendam a página.

Exemplos:

  • Uso de <header>, <main>, <nav> e <footer> para delimitar as regiões da página;
  • Hierarquia de cabeçalhos sequencial e sem saltos (<h1> seguido de <h2>, etc.);
  • Dados tabulares organizados com <table>, <th> e <td>;
  • Listas de elementos estruturadas com <ul> ou <ol>.

Pergunte-se:

  • A estrutura visual da página reflete fielmente a estrutura das tags no código?
  • Um leitor de tela conseguirá identificar onde começa o conteúdo principal?
  • Os níveis de título (<h1> ao <h6>) foram usados para criar hierarquia ou apenas para mudar o tamanho da fonte?
  • As listas e tabelas foram criadas com tags corretas ou apenas simuladas com <div> e CSS?

Se ignorado, há risco de:

  • Desorientação completa de usuários de leitores de tela;
  • Perda da capacidade de navegação rápida por títulos;
  • Falha na interpretação de dados estruturados e tabelas por tecnologias assistivas.

2. Operabilidade e Foco por Teclado​

Todas as funcionalidades e elementos interativos da aplicação devem ser totalmente acessíveis e controláveis sem o uso de mouse.

Exemplos:

  • Links, botões e campos de formulário focáveis nativamente ou com tabindex="0";
  • Indicador de foco (:focus-visible) claramente visível com alto contraste;
  • Link de salto ("Ir para o conteúdo principal") implementado no início do HTML;
  • Ordem de navegação pelo teclado (Tab) seguindo o fluxo visual lógico.

Pergunte-se:

  • É possível navegar, acionar e fechar todos os elementos da página usando apenas o teclado?
  • O usuário que navega por teclado consegue ver exatamente onde o cursor está posicionado na tela?
  • Existe algum elemento que "prende" o foco do teclado, impedindo o usuário de avançar ou retroceder?
  • Links repetitivos de menu podem ser pulados para economizar tempo de digitação?

Se ignorado, há risco de:

  • Bloqueio completo da navegação para usuários de leitores de tela ou com limitações motoras;
  • Navegação "às cegas" pela interface devido à ocultação do indicador de foco no CSS (outline: none);
  • Abandono do site por frustração ou impossibilidade física de concluir tarefas.

3. Acessibilidade em Formulários e Entradas​

Os campos de coleta de dados devem ser fáceis de compreender, identificar e preencher, oferecendo suporte robusto para evitar erros de digitação.

Exemplos:

  • Uso obrigatório da tag <label> associada ao <input> via atributo for;
  • Mensagens de erro em formato de texto conectadas dinamicamente via aria-describedby;
  • Atributo aria-invalid="true" aplicado via JavaScript em campos com erro;
  • Prevenção de redundância (autopreenchimento ou seleção de dados já digitados no mesmo fluxo).

Pergunte-se:

  • Cada campo do formulário possui um rótulo de texto claro que o leitor de tela consiga ler?
  • O sistema avisa onde ocorreu o erro e instrui o usuário textualmente sobre como corrigi-lo?
  • O placeholder foi usado incorretamente como substituto do label (sumindo quando o usuário digita)?
  • O usuário é obrigado a redigitar dados que ele já forneceu minutos atrás no mesmo processo?

Se ignorado, há risco de:

  • Usuários cegos não saberem o que deve ser digitado em um campo de texto;
  • Formulários impossíveis de serem enviados por falhas recorrentes de validação não informadas;
  • Desistência em fluxos complexos (como checkouts de e-commerce) devido ao esforço repetitivo.

4. Robustez e Componentes Dinâmicos (WAI-ARIA)​

Componentes interativos avançados atualizados via JavaScript devem comunicar seus estados, nomes e funções em tempo real para os softwares leitores de tela.

Exemplos:

  • Menus e sanfonas (accordions) alternando dinamicamente aria-expanded="true" e false;
  • Janelas modais recebendo role="dialog" e aria-modal="true" ao serem abertas;
  • Alertas e notificações assíncronas injetadas na tela usando aria-live="polite" ou role="status";
  • Botões customizados (como botões de ícone) possuindo um rótulo textual via aria-label.

Pergunte-se:

  • Quando um menu abre via clique em JavaScript, um usuário cego é avisado programaticamente dessa mudança?
  • O leitor de tela lê o conteúdo de um pop-up ou modal assim que ele surge na tela?
  • Notificações de sucesso ou erro que aparecem sem recarregar a página são anunciadas para quem não enxerga a tela?
  • Os botões que usam apenas ícones (como o "X" de fechar) possuem um nome acessível por trás do código?

Se ignorado, há risco de:

  • Elementos invisíveis ou inacessíveis para tecnologias assistivas, apesar de estarem na tela;
  • Confusão do usuário por falta de feedback sobre interações dinâmicas e assíncronas;
  • Quebra na integridade da interface com elementos flutuantes bloqueando a navegação.

5. Alvos de Toque e Independência de Dispositivo​

Os elementos interativos devem possuir tamanho e espaçamento adequados no código para permitir acionamentos precisos por qualquer tipo de ponteiro (mouse, dedo ou caneta).

Exemplos:

  • Botões e links com área mínima de clique de 24x24 pixels (ou 48x48 pixels para alvos isolados);
  • Uso de padding no CSS para expandir a área clicável de ícones pequenos;
  • Alternativas por clique simples/toque para funções que exigem arrastar elementos (drag-and-drop);
  • Layouts responsivos que não limitam a orientação da tela (funcionam em retrato e paisagem).

Pergunte-se:

  • Elementos clicáveis que ficam lado a lado possuem espaçamento suficiente para evitar cliques acidentais?
  • Links e botões muito pequenos receberam propriedades CSS para aumentar sua área de toque invisível?
  • Um usuário que não consegue fazer movimentos de arrastar consegue operar o sistema por meio de cliques simples?
  • O código bloqueia a rotação da tela em dispositivos móveis, prejudicando quem usa o celular acoplado a uma cadeira de rodas?

Se ignorado, há risco de:

  • Cliques errados frequentes, gerando estresse e acionamentos indesejados;
  • Exclusão de usuários com tremores nas mãos ou limitações de coordenação motora fina;
  • Incompatibilidade da aplicação com diferentes tamanhos de tela e contextos de mobilidade.
